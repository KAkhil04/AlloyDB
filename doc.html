1. Document Overview

This document provides comprehensive architectural and operational documentation for Google AlloyDB Omni Operator version 1.6.0 deployed on Red Hat OpenShift Container Platform version 4.16.46, managing AlloyDB Omni database clusters running version 17.5.0. The operator follows a cluster-scoped deployment model where it is installed into the alloydb-omni-system namespace and manages DBCluster resources and their associated components across multiple tenant namespaces.

AlloyDB Omni is a downloadable, self-managed edition of Google AlloyDB for PostgreSQL. It delivers the high-performance AlloyDB engine—offering more than twice the transactional throughput and up to 100 times faster analytical query performance compared to standard PostgreSQL—packaged as containers managed by a Kubernetes operator. This approach enables automated deployment, lifecycle management, high availability, backup and restore, and monitoring of database instances within enterprise Kubernetes environments.

This document is intended for platform engineers, database administrators, and DevOps teams responsible for deploying, operating, and maintaining AlloyDB Omni on OpenShift. It covers the operator’s internal architecture, the container images it deploys, the Custom Resource Definitions (CRDs) it introduces, the network flows between components, the high availability model, backup and disaster recovery strategies, and the OpenShift-specific considerations that affect deployment and security.

3. AlloyDB Omni Operator Architecture

3.1 Cluster-Scoped Operator Model

The AlloyDB Omni Operator is deployed as a cluster-scoped operator. This means the operator’s controller deployments and associated webhook services run in a single, dedicated namespace (alloydb-omni-system), but the operator watches and manages Custom Resources (CRs) across all namespaces in the cluster. When a user creates a DBCluster resource in any tenant namespace, the operator’s reconciliation loop detects this resource and provisions all the necessary Kubernetes objects—StatefulSets, Services, PersistentVolumeClaims, Secrets, ConfigMaps—in that same tenant namespace.

This cluster-scoped model provides several advantages. A single operator installation serves the entire OpenShift cluster, eliminating the need for per-namespace operator deployments. The operator’s ClusterRole and ClusterRoleBindings grant it the permissions necessary to manage resources across namespace boundaries. The webhook services (fleet-webhook-service and local-webhook-service) validate and mutate CRD resources regardless of which namespace they are created in.

3.2 Operator Namespace (alloydb-omni-system)

The alloydb-omni-system namespace is the operator’s control plane. It contains the following key resources:

  • Fleet Controller Deployment: The primary operator controller that watches for DBCluster and other CRD resources across all namespaces. It runs the global reconciliation logic that translates declarative CRD specifications into Kubernetes-native resources.

  • Local Controller Deployment: A companion controller that handles node-local operations, including direct interactions with database instances running on the same node. The version 1.6.0 release fixed a memory leak in this component.

  • Webhook Services: Admission webhooks (fleet-webhook-service and local-webhook-service) that intercept API server requests to validate and mutate AlloyDB Omni custom resources before they are persisted to etcd. These webhooks enforce schema validation, default values, and cross-resource consistency checks.

  • cert-manager Certificates: TLS certificates issued by cert-manager for securing webhook endpoints and internal operator communications. These include fleet-serving-cert and local-serving-cert, each with DNS names scoped to the alloydb-omni-system namespace.

  • ClusterIssuer: The alloydbomni-selfsigned-cluster-issuer ClusterIssuer resource provides a self-signed certificate authority used to bootstrap trust for the operator’s internal TLS infrastructure.

  • ServiceAccount & RBAC: Dedicated ServiceAccounts with ClusterRole bindings that grant the operator permission to create, read, update, and delete resources across namespaces including Pods, Services, StatefulSets, PVCs, Secrets, ConfigMaps, and all AlloyDB Omni CRDs.

3.3 Tenant Namespace Model

Google recommends creating database clusters in namespaces separate from alloydb-omni-system. This separation improves isolation and security and prevents conflicts between the operator, the resources it manages, and user application components. Each tenant namespace contains the actual database workloads:

  • Database Pods: StatefulSet-managed pods running the AlloyDB Omni database engine along with sidecar containers for log rotation, memory management, and monitoring.

  • Kubernetes Services: ClusterIP or LoadBalancer services providing read-write (RW) and read-only (RO) endpoints for database connectivity.

  • PersistentVolumeClaims: Storage claims for database data directories, WAL logs, and observability data.

  • Secrets: Database admin passwords (sourced from Vault), TLS certificates, Kerberos keytabs, and backup storage credentials.

  • Custom Resources: DBCluster, BackupPlan, Backup, Restore, Failover, PgBouncer, Sidecar, and other CRDs that declare the desired state of the database infrastructure.

4. Operator Components & Container Images

4.1 Operator Controller Pods

The operator deploys two primary controller pods within the alloydb-omni-system namespace. These controllers implement the Kubernetes operator pattern: they run continuous reconciliation loops that compare the desired state declared in CRD resources against the actual state of Kubernetes objects and take corrective action to converge the two.

The fleet controller is responsible for global lifecycle management—creating and scaling StatefulSets, managing Services, handling upgrades, and coordinating multi-instance HA topologies. The local controller handles node-level operations and direct database instance management, including health checks and local configuration updates.

5.2 User-Facing CRDs (alloydbomni.dbadmin.goog)

These 12 CRDs constitute the operator’s public API. Users create and manage these resources in tenant namespaces to declare the desired state of their database infrastructure.

dbclusters

The primary resource that declares an AlloyDB Omni database cluster. A DBCluster encapsulates the complete specification for a database instance including compute resources (CPU, memory), storage disks, database version (17.5.0), control plane agents version, HA configuration, admin user credentials, database parameters, scheduling constraints (node affinity, tolerations, pod anti-affinity), and feature flags (AlloyDB AI, memory agent). The operator watches DBCluster resources across all namespaces and provisions the corresponding StatefulSets, Services, PVCs, Secrets, and ConfigMaps in the tenant namespace.

Key specification fields:

  • databaseVersion: Target database version string (e.g., "17.5.0").

  • controlPlaneAgentsVersion: Version of per-instance control plane agents.

  • primarySpec: Defines resources (CPU/memory), disks, adminUser.passwordRef, parameters (PostgreSQL GUCs), schedulingconfig (nodeaffinity, tolerations, podAntiAffinity), services (Logging, Monitoring), dbLoadBalancerOptions, and sidecarRef.

  • availability: HA settings: numberOfStandbys, enableAutoFailover, enableAutoHeal, autoFailoverTriggerThreshold, autoHealTriggerThreshold, healthcheckPeriodSeconds, enableStandbyAsReadReplica.

  • allowExternalIncomingTraffic: Boolean controlling external service exposure.

  • mode: Cluster mode for DR scenarios (primary/standby).

  • connectedModeSpec: Optional GCP project binding for connected-mode features.

dbinstances

Represents an individual database instance within a cluster. While DBCluster is the high-level declaration, DBInstance provides visibility into individual instance state. In an HA cluster with one primary and two standbys, there would be three DBInstance resources. This CRD gives administrators per-instance control and observability, including the ability to inspect instance-level health, role (primary/standby), and configuration.

backupplans

Defines the continuous backup configuration for a DBCluster. Specifies the backup storage location (local PV, GCS via gcsOptions, or S3-compatible via s3Options), cron-based backup schedules for full, incremental, and differential backups, a retention period in days, and a paused flag to suspend automatic backups. When no schedule is defined, the default is a daily full backup at midnight. In this environment, backups target NetApp StorageGRID via the s3Options field.

backups

Represents an individual backup, either scheduled by a BackupPlan or created manually. References a DBCluster and a BackupPlan. Manual backups set manual: true and can specify physicalBackupSpec.backupType as "full", "diff", or "incr". The status subresource tracks completion time, backup phase, and any critical incidents.

restores

Triggers restoration of a database cluster from a backup. Supports two modes: restoring from a named backup (backup field), or point-in-time recovery to a specific timestamp (pointInTime field) using continuous WAL archives. The clonedDBClusterConfig field enables restoring into a new DBCluster, allowing non-destructive recovery validation without affecting the source cluster.

failovers

Triggers an unplanned failover, promoting a standby instance to primary when the current primary has become unavailable. Created in the tenant namespace with a dbclusterRef pointing to the target cluster. The operator executes the promotion, reconfigures replication topology, and updates service endpoints. Failover metrics are exposed via kube-state-metrics with the alloydb_omni_failover_ prefix.

switchovers

Triggers a planned, controlled switchover between primary and standby instances. Unlike a failover (which handles unplanned outages), a switchover is used for planned maintenance such as node patching, storage migration, or load rebalancing. The switchover process demotes the current primary gracefully, ensures all WAL records are replicated, and promotes the target standby—resulting in zero data loss and minimal downtime.

pgbouncers

Deploys a PgBouncer connection pooling proxy for a DBCluster. Defines the target cluster via dbclusterRef, replica count, pool configuration parameters (pool_mode, default_pool_size, max_client_conn, max_db_connections, ignore_startup_parameters), pod resource requirements, container image override (must point to Artifactory), scheduling config (node affinity), and service type (ClusterIP or LoadBalancer).

replications

Configures cross-cluster or cross-region replication between a primary DBCluster and a secondary (standby) DBCluster. Used for disaster recovery setups where the primary cluster in one region asynchronously streams WAL records to a secondary cluster in a remote region or data center.

sidecars

Attaches custom sidecar containers to database pods. The sidecar specification defines one or more containers with their images, commands, arguments, resource limits, and volume mounts. Sidecars can mount the obsdisk volume to access database diagnostic logs at /obs/diagnostic/. Common use cases include Splunk log forwarding, custom monitoring agents, or enterprise backup tool integration. Registered to a DBCluster via primarySpec.sidecarRef.

tdeconfigs

Configures Transparent Data Encryption (TDE) for a database cluster. TDE encrypts data at rest at the storage layer, ensuring that the physical database files on disk are encrypted without requiring application-level changes. This CRD manages encryption key configuration and integration with key management systems for key rotation and lifecycle management.

userdefinedauthentications

Configures custom authentication mechanisms for the database cluster beyond the default pg_hba.conf settings. In this environment, this CRD is used to integrate Kerberos/GSSAPI authentication rules, enabling enterprise SSO through Active Directory and Kerberos keytab-based service principal authentication.

5.4 CRD Relationship Map

The AlloyDB Omni CRDs form a hierarchical relationship model. A DBCluster (user-facing) is decomposed by the operator into one or more instances (internal), each representing a database pod. When HA is configured, the operator creates createstandbyjobs (internal) to provision standby instances. A BackupPlan (user-facing) is decomposed into instancebackupplans (internal), and each Backup (user-facing) spawns instancebackups (internal) for execution. The same pattern applies to Restores, Failovers, Switchovers, Sidecars, and UserDefinedAuthentications.

This two-tier architecture (user-facing declaration + internal execution) allows the operator to present a simple, cluster-level API to users while managing the complex, instance-level orchestration internally. It also provides granular troubleshooting capability: if a backup appears stuck at the DBCluster level, administrators can inspect the corresponding instancebackups resources to identify which specific instance is failing and at which step.

6. Resource Lifecycle & Reconciliation

The AlloyDB Omni Operator follows the standard Kubernetes operator pattern of continuous reconciliation. The operator’s controllers watch for changes to AlloyDB Omni CRDs and the Kubernetes resources they manage. When a discrepancy is detected between the desired state (as declared in the CRD spec) and the actual state (as observed in the cluster), the operator takes corrective action.

Provisioning Flow

When a user applies a DBCluster manifest to a tenant namespace, the operator executes the following sequence:

  1. The fleet webhook validates the DBCluster specification against schema constraints and cross-resource dependencies.

  2. The fleet controller detects the new DBCluster resource and begins reconciliation.

  3. PersistentVolumeClaims are created for the data disk, WAL disk, and observability disk based on the spec.

  4. A StatefulSet is created with the database pod template, including the database container, dbinit init container, and all sidecar containers (logrotate-agent, memoryagent, alloydb-monitor).

  5. Kubernetes Services are created: a read-write (RW) service pointing to the primary instance and a read-only (RO) service for replica traffic.

  6. The dbinit container runs initdb to initialize the PostgreSQL data directory on the persistent volume.

  7. The database container starts and the control plane agents configure the instance according to the DBCluster spec.

  8. The operator updates the DBCluster status to reflect DBClusterReady when all components are healthy.

Upgrade & Update Flow

Database version upgrades and operator version updates follow controlled rolling update semantics. For minor version upgrades, the operator updates the database container image and triggers a controlled restart. For major version upgrades (e.g., 15.x to 17.x), the process may require manual pg_upgrade steps. When upgrading HA clusters, the operator may require temporarily disabling HA (setting numberOfStandbys to 0), performing the upgrade, and re-enabling HA afterward. Starting with operator 1.6.0, modifications to scheduling configuration are applied immediately and trigger a database restart.

7.2 Kubernetes Services

The operator creates two primary services for each DBCluster in the tenant namespace:

  • Read-Write (RW) Service: Named as <prefix>-<dbcluster>-rw-ilb, this ClusterIP service routes traffic exclusively to the current primary database instance. It uses label selectors including dbs.internal.dbadmin.goog/ha-role=Primary to ensure traffic reaches only the active primary.

  • Read-Only (RO) Service: Named as <prefix>-<dbcluster>-ro-ilb, this service distributes read traffic across standby replicas when HA is configured with enableStandbyAsReadReplica set to true.

Starting with operator 1.3.0, read-only and read-write load balancers are created concurrently during new cluster provisioning, reducing the time until the cluster is ready for connections.

7.3 PgBouncer Connection Pooling

When a PgBouncer CRD is applied, the operator creates a separate Deployment running PgBouncer pods. These pods connect to the database through the RW service and expose their own Service endpoint for client connections. PgBouncer supports multiple pool modes (session, transaction, statement) and provides a statistics interface accessible via the pgbouncer virtual database using the statsuser account.

7.4 TLS & Certificate Management

The operator relies on cert-manager to issue and manage TLS certificates for internal communications. cert-manager is already available on the cluster, integrated with the enterprise Venafi solution for automated certificate provisioning and renewal. The operator creates the following cert-manager resources in the alloydb-omni-system namespace:

  • ClusterIssuer (alloydbomni-selfsigned-cluster-issuer): A cluster-wide self-signed certificate issuer used as the root of trust.

  • Namespace Issuers: fleet-selfsigned-issuer and local-selfsigned-issuer for webhook certificate issuance.

  • Certificates: fleet-serving-cert and local-serving-cert providing TLS for webhook HTTPS endpoints.

Database client connections use TLS (TLSv1.3 with AES-256-GCM-SHA384) by default.

8. High Availability Architecture

8.1 Patroni-Based HA

AlloyDB Omni implements high availability using Patroni, an open-source cluster management tool for PostgreSQL. Patroni automates the setup, monitoring, and failover of PostgreSQL clusters by coordinating instance roles through a distributed consensus system. In the Kubernetes deployment model, Patroni uses the Kubernetes API itself as its distributed configuration store (DCS), eliminating the need for an external etcd or ZooKeeper cluster.

8.2 Streaming Replication

AlloyDB Omni uses standard PostgreSQL streaming replication to maintain data consistency between primary and standby instances:

  • Synchronous Replication (default): Each transaction committed on the primary is synchronously replicated to standby instances before the commit is acknowledged to the client. This provides RPO of zero—no data loss during failover. The trade-off is increased transaction latency.

  • Asynchronous Replication: WAL records are sent to standbys asynchronously, reducing write latency but introducing potential data loss. Typically used for cross-region disaster recovery where network latency makes synchronous replication impractical.

The primary maintains a replication slot for each standby instance. Replication status can be verified through pg_replication_slots and pg_stat_replication system views.

8.3 Automatic Failover

When HA is enabled through the DBCluster’s availability section, the operator configures automatic failover behavior through:

  • enableAutoFailover: Enables or disables automatic failover when the primary becomes unresponsive.

  • autoFailoverTriggerThreshold: The number of consecutive health check failures before failover is triggered (e.g., 3 failures).

  • healthcheckPeriodSeconds: The interval between health checks (default 30 seconds, range 1–86400).

  • enableAutoHeal: Enables automatic recovery of failed instances without operator intervention.

When Patroni detects primary failure, the remaining standbys coordinate to select the most suitable replica for promotion. The selected standby replays outstanding WAL records, promotes itself, and updates Kubernetes service endpoints. This process typically achieves an RTO of less than sixty seconds.

9. Backup & Disaster Recovery

AlloyDB Omni provides a comprehensive backup and recovery system managed entirely through Kubernetes CRDs. In this environment, all backups are stored on NetApp StorageGRID, an on-premises S3-compatible object storage platform.

Continuous Backup

When a BackupPlan resource is created for a DBCluster, the operator enables continuous backup. The system archives PostgreSQL Write-Ahead Log (WAL) files to the configured StorageGRID bucket, enabling point-in-time recovery (PITR) to any second within the retention window.

StorageGRID S3 Integration

In this disconnected environment, backup storage uses NetApp StorageGRID via the S3-compatible API. The BackupPlan CRD is configured with s3Options pointing to the internal StorageGRID endpoint:

  • S3 Credentials Secret: A Kubernetes Secret containing access-key-id and access-key fields, created in the tenant namespace. These credentials are provisioned through the DB Ops HashiCorp Vault and injected as Kubernetes Secrets via the Vault Secrets Operator.

  • StorageGRID Endpoint: The internal StorageGRID S3 API endpoint URL configured in s3Options.endpoint. The CA bundle must be provided via s3Options.caBundle or a certRef secret to establish trust with internal TLS certificates.

  • Bucket & Key Path: Each DBCluster’s backups are isolated by bucket or key prefix path, ensuring multi-tenancy at the storage level.

Point-in-Time Recovery

The Restore CRD supports PITR by specifying a pointInTime timestamp along with the sourceDBCluster. The operator uses the continuous WAL archive on StorageGRID to reconstruct the database state at the exact requested timestamp.

Cross-Region Disaster Recovery

For cross-region DR, AlloyDB Omni supports asynchronous streaming replication from a primary DBCluster in one region to a secondary DBCluster in a remote region. The secondary can be promoted via the Failover CRD. StorageGRID’s native cross-site replication can also ensure backup data availability across data centers independently of database replication.

10. Monitoring & Observability

Database-Level Metrics

The alloydb-monitor sidecar container exposes PostgreSQL and AlloyDB-specific metrics in Prometheus format:

  • alloydb_omni_memory_used_byte: Memory consumption of the database container in bytes.

  • alloydb_omni_instance_postgresql_wait_time_us_total: Total wait time across PostgreSQL wait event categories.

  • Standard PostgreSQL metrics: Connection counts, transaction rates, cache hit ratios, replication lag, WAL generation rates, lock statistics.

Custom Resource Metrics

Starting with operator version 1.3.0, the operator exposes custom resource metrics through kube-state-metrics:

  • DBCluster metrics (alloydb_omni_dbcluster_*): HA readiness, primary node readiness, database version, cluster phase, resource utilization.

  • Failover metrics (alloydb_omni_failover_*): Failover event counts, duration, and status.

  • PgBouncer metrics: Connection pool utilization, client and server connection counts, query routing statistics.

  • Backup/BackupPlan metrics: Backup completion status, timing, and storage utilization (new in operator 1.6.0).

Splunk Log Aggregation

Database diagnostic logs are forwarded to Splunk for centralized log management, search, and alerting. The following log streams are captured:

  • PostgreSQL Server Logs (postgresql.log): All database server activity including connection events, query errors, slow queries, checkpoint activity, and autovacuum operations.

  • Audit Logs (postgresql.audit): Session and object-level audit trails generated by the pgaudit extension. Captures DDL, DML on sensitive tables, and role/privilege changes.

  • Operator Controller Logs: Fleet and local controller logs from alloydb-omni-system, capturing reconciliation events, errors, and lifecycle operations.

The logrotate-agent sidecar manages on-pod rotation with configurable thresholds, gzip compression, and 7-day local retention. The Splunk forwarder sidecar operates independently, streaming logs in real-time before rotation occurs.

Dynatrace Application Performance Monitoring

Dynatrace integration for AlloyDB Omni is currently being developed in collaboration with Google. The specifics (sidecar injection method, metric mapping, dashboard templates) are subject to the outcome of the ongoing engagement and will be updated in a future revision.

The target architecture involves deploying the Dynatrace OneAgent as a sidecar or leveraging the Dynatrace Operator’s code injection for infrastructure monitoring (CPU, memory, disk I/O), database performance analysis (query-level, connection pools, lock contention), and end-to-end distributed tracing.

11.2 Disconnected Environment & Image Pipeline

This OpenShift cluster operates in a fully air-gapped environment. All container images must be pre-staged in the internal Artifactory container registry before deployment.

Quarantine Pipeline

  1. Image Request: The platform team identifies required images and versions from the bank-compliant Helm chart provided by the Google vendor team.

  2. ATAAS Quarantine Pull: An ATAAS (Automation-Testing-as-a-Service) based quarantine job running in a controlled DMZ pulls images from gcr.io/alloydb-omni/ through an approved egress proxy.

  3. Aqua Security Scanning: Images undergo Aqua container security scanning including CVE analysis, malware detection, compliance validation against CIS benchmarks, and license risk assessment.

  4. Approval & Promotion: Images that pass all Aqua scanning gates are promoted from the ATAAS quarantine staging area to the production Artifactory container registry.

  5. Registry Path Mapping: Images are published under an internal path (e.g., artifactory.internal.company.com/alloydb-omni/) mirroring upstream structure.

When installing via Helm (rather than OLM), administrators must explicitly bind the operator’s ServiceAccount to the appropriate SCC (typically anyuid or a custom SCC) through a ClusterRoleBinding, as Helm does not automatically negotiate SCC bindings.

UBI-based images pass Red Hat’s container certification requirements and are compatible with cgroup v2 and Linux kernel 4.18+.

13. Security Architecture

The AlloyDB Omni Operator implements a defense-in-depth security architecture. In this environment, the security model is enhanced with enterprise Kerberos authentication, HashiCorp Vault for secrets management, and comprehensive audit logging forwarded to Splunk.

13.1 Network Security

  • TLS Encryption: All database connections use TLS 1.3 by default (TLS_AES_256_GCM_SHA384 cipher). Webhook communications use cert-manager TLS certificates. Inter-pod replication traffic is also encrypted.

  • Network Policies: OpenShift’s SDN enforces NetworkPolicy resources to restrict traffic to database pods from only authorized namespaces or pod selectors.

  • Service Exposure Control: The allowExternalIncomingTraffic flag controls whether database services are accessible from outside the cluster.

  • Air-Gap Boundary: The disconnected environment ensures no database traffic can egress to the internet. All images, certificates, and configuration are pre-staged internally.

13.2 Kerberos Authentication & Keytab Integration

Database client authentication leverages the enterprise Kerberos infrastructure for single sign-on (SSO) capability. The integration works through the UserDefinedAuthentication CRD and direct PostgreSQL pg_hba.conf configuration managed by the operator.

  • Kerberos Principal Mapping: Database users are mapped to Kerberos principals from the enterprise Active Directory or MIT Kerberos KDC. The pg_hba.conf is configured with GSSAPI authentication entries.

  • Keytab File Provisioning: Kerberos keytab files containing the database service principal’s long-term keys are provisioned as Kubernetes Secrets (from Vault) and mounted into database pods. The keytab allows PostgreSQL to authenticate to the KDC without interactive password entry.

  • pg_hba.conf Configuration: The operator applies Kerberos authentication rules via the UserDefinedAuthentication CRD, adding GSSAPI entries that specify the Kerberos realm, keytab file path, and user mapping rules.

  • Client Requirements: Client applications must obtain a valid Kerberos ticket (via kinit or SSPI) before connecting. The PostgreSQL libpq client handles GSSAPI/Kerberos negotiation transparently.

13.3 HashiCorp Vault Integration (VPM Model)

All tenant namespaces use the HashiCorp Vault VPM (Vault Platform Model) for centralized secrets management. Database operational secrets—including admin passwords, backup storage credentials, TLS certificates, and Kerberos keytabs—are stored in the DB Ops Vault path and injected into Kubernetes Secrets:

  • Vault Secrets Operator (VSO): Watches VaultStaticSecret or VaultDynamicSecret CRs in tenant namespaces and synchronizes referenced Vault secrets into native Kubernetes Secrets. Ensures DBCluster’s adminUser.passwordRef, backup S3 credentials, and TLS secrets stay in sync with Vault.

  • Vault Agent Sidecar Injection: Alternative pattern where a Vault Agent init container retrieves secrets at pod startup and renders them into files or environment variables. Used for Kerberos keytab files that need mounting at specific container paths.

  • Vault Policy Scoping: Each tenant namespace’s ServiceAccount is bound to a Vault role with a policy scoped to only the DB Ops secrets path relevant to that namespace. Strict multi-tenant isolation ensures a namespace can only access its own database secrets.

Secrets lifecycle: Secrets are created and rotated in Vault by the DB Ops team → Vault Secrets Operator synchronizes them to Kubernetes Secrets in the tenant namespace → AlloyDB Omni Operator references these Kubernetes Secrets in DBCluster, BackupPlan, and PgBouncer specs. Password rotation in Vault automatically propagates through this chain.

13.4 Container Security & Audit

  • UBI 9 Base Images: All containers use Red Hat UBI 9 with regular CVE patches. Images are scanned with Aqua Security in the ATAAS-based Quarantine pipeline before promotion to Artifactory.

  • OpenShift SCC Compliance: The openshift.io/scc=anyuid annotation ensures predictable UID behavior. The operator’s ServiceAccount is explicitly bound to the required SCC.

  • Image Provenance: All images are pulled exclusively from Artifactory. No external registry access is possible. All images pass Aqua Security scanning in the ATAAS quarantine before reaching production, eliminating supply chain attack vectors.

  • PostgreSQL Audit Logging: The pgaudit extension provides session and object-level audit trails written to postgresql.audit and forwarded to Splunk.

  • Kubernetes Audit: OpenShift API server audit logs capture all oc operations against AlloyDB CRDs.

  • Vault Audit: All Vault secret access operations are logged in Vault’s audit backend, providing traceability of which service accounts accessed which database secrets.
